Static analysis tools differ in how deeply they track data through code. A tool that only matches syntax will catch the textbook case but miss the same vulnerability once it moves behind a local variable, a method call, or a builder chain. We tested **Semgrep**, **CodeQL**, and **OpenTaint** on the same intentionally vulnerable Spring Boot application to show exactly where each tool's analysis model hits its limit.

## Tools

### Semgrep

Semgrep is a static analysis tool built around pattern matching, with taint-analysis support and broader inter-procedural coverage in commercial editions. Its main strength is approachable rule authoring across multiple programming languages. The open-source version is referred to as "Semgrep" below, with CE and Code distinguished when relevant.

### CodeQL

CodeQL is GitHub's semantic static analysis engine. It exposes a dedicated query language and strong variant-analysis workflow, with support for multiple languages. It is free for open-source repositories and available for private repositories through GitHub Advanced Security.

### OpenTaint

OpenTaint is an open-source static analysis tool that interprets Semgrep-style patterns as dataflow queries rather than syntactic matches. Metavariables are tracked as program values through assignments, method calls, field chains, and virtual dispatch. It supports Java and Kotlin, with Python, Go, C#, and JavaScript planned. The core engine is licensed under Apache 2.0; the CLI, CI integrations, and rules under MIT.

## Methodology

The test application is an intentionally vulnerable Spring Boot project designed to isolate different aspects of XSS detection. Each example measures two things:

- XSS vulnerabilities that the tool fails to detect (false negatives).
- Secure code incorrectly flagged as vulnerable (false positives).

Semgrep and OpenTaint use custom rules; CodeQL uses its default `java/xss` rule.

## Five levels of analysis depth

XSS is well-understood. What varies is how much code complexity a tool can see through to find it.

The five test cases form a progression of analytical capabilities, each demanding something the previous one did not:

| Level | Capability required | What the code does |
|---|---|---|
| 1 | Syntax matching | User input returned directly |
| 2 | Local dataflow | Value assigned to variable before return |
| 3 | Inter-procedural analysis | Value flows through a helper method |
| 4 | Field sensitivity | Value passes through constructor chains and nested objects |
| 5 | Pointer analysis | Value flows through builder pattern with virtual dispatch |

Each level reflects patterns that are routine in production code. The question is not whether XSS is dangerous — it is which of these ordinary coding patterns cause a tool to lose track of the data.

### Level 1: Syntax matching — direct return

The baseline: user input flows directly into an HTML response. No indirection, no intermediate variables. Any tool that understands source-to-sink should catch this.

```java
// Display user profile with custom message
@GetMapping("/profile/display")
@ResponseBody
public String displayUserProfile(
        @RequestParam(defaultValue = "Welcome") String message) {
    // Direct output without escaping
    return "<html><body><h1>Profile Message: " + message + "</h1></body></html>";
}
```

If a user visits `/profile/display?message=<script>alert('XSS')</script>`, the application returns executable script content to the browser.

A Semgrep rule to detect this pattern:

```yaml
id: pattern.xss
patterns:
  - pattern: |
      @$ANNOTATION(...)
      $RETURNTYPE $METHOD(..., $TYPE $PARAM,...) {
        ...
        return "..." + $PARAM + "...";
        ...
      }
```

This rule triggers when a method annotated with `$ANNOTATION` returns a concatenation of `$PARAM` with constant strings, indicating a potential XSS vulnerability.

All three tools detect this case. No surprise — it is the minimal XSS shape.

**Results**: ✅ **Semgrep**, ✅ **CodeQL**, ✅ **OpenTaint**

### Level 2: Local dataflow — variable assignment

Production code almost never returns user input directly. The first thing a developer does is assign it to a variable — and that single step is enough to break syntax-only matching. A tool needs local dataflow tracking to follow the value from assignment to return.

```java
// Display user status with local variable assignment
@GetMapping("/profile/status")
@ResponseBody
public String displayUserStatus(
        @RequestParam(defaultValue = "Active") String message) {
    // Assign to local variable
    String statusMessage = "<html><body><h1>User Status: " + message + "</h1></body></html>";
    return statusMessage;
}
```

The vulnerable value is first assigned to `statusMessage` and then returned. The pattern rule from Level 1 no longer matches because the return statement contains a variable, not a concatenation.

OpenTaint catches this with a simpler rule than Level 1 required — because the engine treats the pattern as a dataflow query, not a syntax match:

```yaml
id: pattern.xss
patterns:
  - pattern: |
      @$ANNOTATION(...)
      $RETURNTYPE $METHOD(..., $TYPE $PARAM,...) {
        ...
        return $PARAM;
        ...
      }
```

This rule captures the core condition: user input reaches a return statement. OpenTaint tracks `$PARAM` through the assignment and finds it in `statusMessage`.

Semgrep can handle this scenario too, but requires switching to a taint rule with explicit source and sink definitions:

```yaml
id: taint.xss
mode: taint
pattern-sources:
  - patterns:
      - focus-metavariable: $PARAM
      - pattern: |
          @$ANNOTATION(...)
          $RETURNTYPE $METHOD(..., $TYPE $PARAM,...) {
            ...
          }
pattern-sinks:
  - patterns:
      - focus-metavariable: $SINK
      - pattern: |
          @$ANNOTATION(...)
          $RETURNTYPE $METHOD(...) {
            ...
            return $SINK;
            ...
          }
```

Both approaches work. The difference is how much of the dataflow model has to be expressed in the rule itself. In OpenTaint, the engine infers the flow; in Semgrep, the rule author declares it.

#### Handling sanitization

The secure version escapes user input before including it in the response:

```java
// Display escaped user status with local variable assignment
@GetMapping("/profile/secureStatus")
@ResponseBody
public String displaySecureUserStatus(
        @RequestParam(defaultValue = "Active") String message) {
    // Assign to local variable with proper escaping
    String htmlContent = "<html><body><h1>User Status: " +
            HtmlUtils.htmlEscape(message) + "</h1></body></html>";
    return htmlContent;
}
```

The basic rules above still flag this as vulnerable because they don't recognize the sanitization function. To handle it, enhance the pattern rule with a negative pattern:

```yaml
patterns:
  - pattern: |
      @$ANNOTATION(...)
      $RETURNTYPE $METHOD(..., $TYPE $PARAM,...) {
        ...
        return $PARAM;
        ...
      }
  - pattern-not: |
      @$ANNOTATION(...)
      $RETURNTYPE $METHOD(..., $TYPE $PARAM,...) {
        ...
        return HtmlUtils.htmlEscape($PARAM);
        ...
      }
```

Taint rules use sanitizer patterns instead:

```yaml
pattern-sanitizers:
  - pattern: HtmlUtils.htmlEscape(...)
```

Sanitizers tell the analysis engine to stop tracking tainted data when it passes through the specified function.

**Results**:

- ❌ **Semgrep (pattern)**: Misses the vulnerability — syntax matching cannot follow the assignment.
- ✅ **Semgrep (taint)**: Detects the vulnerability and can recognize sanitization.
- ✅ **CodeQL** and ✅ **OpenTaint (pattern and taint)**: Correctly handle both vulnerable and secure code.

From this point forward, Semgrep's taint rules are used (pattern rules are insufficient), with OpenTaint's results shown for both rule types.

### Level 3: Inter-procedural analysis — function call boundary

The vulnerable value now moves through a helper method before reaching the response. The tool must follow data across function boundaries — not just within a single method.

```java
// Generate user dashboard with custom greeting
@GetMapping("/dashboard/greeting")
@ResponseBody
public String generateDashboard(
        @RequestParam(defaultValue = "Welcome") String greeting) {
    String htmlContent = buildDashboardContent(greeting);
    return htmlContent;
}

private static String buildDashboardContent(String greeting) {
    // Generate dashboard HTML content
    return "<html><body><h1>Dashboard: " + greeting + "</h1></body></html>";
}
```

The secure version escapes inside the helper:

```java
// Generate user dashboard with custom greeting
@GetMapping("/dashboard/secureGreeting")
@ResponseBody
public String generateSecureDashboard(
        @RequestParam(defaultValue = "Welcome") String greeting) {
    String htmlContent = buildSecureDashboardContent(greeting);
    return htmlContent;
}

private static String buildSecureDashboardContent(String greeting) {
    // Generate dashboard HTML content with escaped greeting
    return "<html><body><h1>Dashboard: " +
            HtmlUtils.htmlEscape(greeting) + "</h1></body></html>";
}
```

This is the first case where the tools separate clearly. Semgrep CE does not model the sanitizing callee precisely enough to suppress the finding on the secure version, producing a false positive. Semgrep Code handles it correctly because its inter-procedural analysis inspects what happens inside the called method.

**Results**:

- ❌ **Semgrep CE**: Produces false positives — cannot see sanitization inside the callee.
- ✅ **Semgrep Code**: Correctly handles both vulnerable and secure code.
- ✅ **CodeQL** and ✅ **OpenTaint**: Correctly handle both vulnerable and secure code.

From this point, Semgrep Code is used for remaining examples since inter-procedural analysis is essential.

### Level 4: Field sensitivity — constructor chains

Now the data flows through nested objects, constructor chains, and field access patterns common in production code. The tool must track a value as it is stored into a field, wrapped in a constructor, and accessed through a chain of property reads.

```java
// Generate message template
@GetMapping("/notifications/template")
@ResponseBody
public String generateTemplate(
        @RequestParam(defaultValue = "New Message") String content) {
    Profile.MessageTemplate template = new Profile.MessageTemplate(content);
    // Return nested content
    return template.body.content.text;
}
```

The HTML content is built through a chain of constructors:

```java
public MessageTemplate(String text) {
    this.body = new MessageBody("<html>" + text + "</html>");
}

public MessageBody(String text) {
    this.content = new MessageContent("<body>" + text + "</body>");
}

public MessageContent(String text) {
    this.text = "<h1>Notification: " + text + "</h1>";
}
```

All three tools detect this first constructor-based example. The next variant extends the field chain further:

```java
// Generate user notification with complex data structure
@GetMapping("/notifications/generate")
@ResponseBody
public String generateNotification(
        @RequestParam(defaultValue = "New Message") String content) {
    // Create user profile with nested message structure using constructors
    Profile.UserProfile profile = new Profile.UserProfile(content);

    // Return deeply nested content
    return profile.settings.config.template.body.content.text;
}
```

Here the tools diverge. Semgrep Code and OpenTaint track the deeper field chain. CodeQL does not report the vulnerability — a common static-analysis tradeoff where following data through complex object aggregation is expensive, and tools make different precision and performance choices.

**Results**:

- ✅ **Semgrep Code**: Detects both simple and complex versions.
- ⚠️ **CodeQL**: Handles the simple version but misses the complex one.
- ✅ **OpenTaint**: Correctly handles both versions, including secure variants.

### Level 5: Pointer analysis — builder pattern with virtual dispatch

Builder patterns are common in modern Java and create some of the hardest dataflow scenarios for static analysis. The tool must track data through method chaining, field assignment, and object state updates.

```java
// Display custom message
@GetMapping("/message/display")
@ResponseBody
public String displayMessage(
        @RequestParam(defaultValue = "Welcome") String message) {
    // Construct a page using a chain of builders
    String page = new HtmlPageBuilder().message(message).buildPage();
    return page;
}
```

The builder class stores the message in a field and constructs the response:

```java
private String message = "";

public HtmlPageBuilder message(String message) {
    this.message = message;
    return this;
}

public String buildPage() {
    return "<html><body><h1>Profile Message: " + message + "</h1></body></html>";
}
```

CodeQL and OpenTaint detect the vulnerability. Semgrep Code does not — builder patterns combine method chaining, field assignment, and object state, which exceeds its current analysis model.

The next variant adds an interface-based formatter:

```java
// Display formatted message
@GetMapping("/message/format")
@ResponseBody
public String formatMessage(
        @RequestParam(defaultValue = "Welcome") String message) {
    // Construct a page using a formatter interface
    String page = new HtmlPageBuilder().message(message)
            .format(new DefaultFormatter()).buildPage();
    return page;
}
```

The formatter interface adds another analytical requirement — virtual dispatch resolution:

```java
public HtmlPageBuilder format(IFormatter formatter) {
    this.message = formatter.format(this.message);
    return this;
}

public class DefaultFormatter implements IFormatter {
    @Override
    public String format(String value) {
        return value; // No actual formatting
    }
}
```

The analyzer must resolve which implementation of `format()` executes at runtime and continue the dataflow through that call. This depends on pointer analysis: determining what concrete objects a reference may point to.

Semgrep Code and CodeQL miss the interface-based case. OpenTaint reports it.

The secure version uses an `EscapeFormatter` that sanitizes the input:

```java
// Display escaped message
@GetMapping("/message/escape")
@ResponseBody
public String escapeMessage(
        @RequestParam(defaultValue = "Welcome") String message) {
    // Construct a page using a formatter as a parameter for a chain of builders
    String page = new HtmlPageBuilder().message(message)
            .format(new EscapeFormatter()).buildPage();
    return page;
}
```

This requires the analyzer to resolve the virtual method call to `EscapeFormatter.format()` and recognize it as a sanitizer. OpenTaint correctly identifies this as safe.

**Results**:

- ❌ **Semgrep Code**: Misses both builder variants.
- ⚠️ **CodeQL**: Handles the simple builder but misses the interface-based version.
- ✅ **OpenTaint**: Detects both patterns, resolves virtual dispatch, and correctly filters the secure `EscapeFormatter` variant.

## What the progression shows

Each tool plateaus at a different level of analytical depth:

- **Semgrep CE** handles syntax matching and local taint tracking (levels 1–2), but stops at function boundaries.
- **Semgrep Code** extends through inter-procedural analysis and field sensitivity (levels 3–4), but does not follow builder patterns or virtual dispatch.
- **CodeQL** covers most cases through level 4, but its analysis depth limits surface at deep field chains and virtual calls.
- **OpenTaint** tracks data through all five levels — including builder state, constructor chains, and interface dispatch — using the same pattern rules throughout.

The key design difference: in Semgrep, the rule author must declare the dataflow model — sources, sinks, sanitizers. In OpenTaint, the engine infers it. A pattern that mentions a parameter and a return statement is enough for the engine to find every path between them, regardless of how many assignments, method calls, or object boundaries the data crosses.

This is not a claim that one approach is universally better. Explicit taint rules give fine-grained control. Implicit dataflow inference reduces rule complexity but depends on the engine's analysis depth. The test cases above show where that depth matters.
